# RSSAggregator

A command-line RSS feed aggregator written in Go. It lets multiple users register, add RSS feeds, follow them, and browse aggregated posts stored in a PostgreSQL database. A background ticker continuously scrapes feeds on a configurable interval, so your reading list stays up to date without any manual action.

## Overview

RSSAggregator is a backend CLI tool. There is no web interface. All interaction happens through subcommands passed as arguments to the compiled binary. The program reads a JSON config file from your home directory to know which database to connect to and which user is currently active. Multiple users can share the same database; each user manages their own set of followed feeds and reads their own posts.

The aggregation loop (`agg`) runs indefinitely and uses a least-recently-fetched strategy to decide which feed to scrape next, so no single feed starves out others.

---

## Architecture

```
main.go              -- Entry point. Reads config, opens DB connection, registers all commands.
commands.go          -- Command dispatcher. Maps command name strings to handler functions.
handler_user.go      -- Handlers for user registration, login, listing, and reset.
handler_feed.go      -- Handlers for adding feeds and listing all feeds in the database.
handler_follow.go    -- Handlers for following, listing follows, and unfollowing feeds.
aggregation.go       -- Scraping loop and browse command.
rss.go               -- HTTP fetch and XML parsing of RSS 2.0 feeds.

internal/config/     -- Reads and writes ~/.gatorconfig.json.
internal/database/   -- Type-safe query layer generated by sqlc from sql/queries/.

sql/schema/          -- Goose migration files (applied in order).
sql/queries/         -- Raw SQL queries that sqlc compiles into Go.
sqlc.yaml            -- sqlc configuration pointing schema and queries at internal/database.
```

The application does not use an HTTP server or any framework. Commands are dispatched by matching `os.Args[1]` against a registry of handler functions. Every handler receives a `*state` struct that holds the database query object and the current config.

```bash
go install github.com/pressly/goose/v3/cmd/goose@latest
```


## Installation

Clone the repository and build the binary:

```bash
git clone https://github.com/Eng-Moaz/RSSAggregator.git
cd RSSAggregator
go build -o gator .
```

This produces a single binary named `gator`. You can move it anywhere on your `$PATH`:

```bash
mv gator /usr/local/bin/gator
```

## Configuration

The program reads `~/.gatorconfig.json` on every invocation. Create this file manually before first use:

```json
{
  "db_url": "postgres://username:password@localhost:5432/gator?sslmode=disable",
  "current_user_name": ""
}
```

| Field | Description |
|-------|-------------|
| `db_url` | Full PostgreSQL connection string. |
| `current_user_name` | The name of the currently active user. Updated automatically by `login` and `register`. |

The `current_user_name` field is rewritten by the program whenever you run `login` or `register`. You do not need to edit it by hand after setup.

## Database Setup

Create a PostgreSQL database:

```bash
createdb gator
```

Apply all migrations using Goose from the project root:

```bash
goose -dir sql/schema postgres "postgres://username:password@localhost:5432/gator?sslmode=disable" up
```

This applies five migrations in order:

1. Creates the `users` table.
2. Creates the `feeds` table with a foreign key to `users`.
3. Creates the `feed_follows` join table with a unique constraint per user/feed pair.
4. Adds the `last_fetched_at` column to `feeds` (used to track scraping order).
5. Creates the `posts` table with a foreign key to `feeds`.

## Usage

All commands follow the same pattern:

```
gator <command> [arguments...]
```

### User Management

**Register a new user**

Creates a new user in the database and sets them as the current user in the config.

```bash
gator register <username>
```

**Switch the active user**

Changes `current_user_name` in the config. The user must already exist in the database.

```bash
gator login <username>
```

**List all registered users**

Prints all usernames. The currently active user is marked with `(current)`.

```bash
gator users
```

**Reset the users table**

Deletes all users (and all cascading data). Intended for development and testing.

```bash
gator reset
```

---

### Feed Management

**Add a feed**

Adds a new RSS feed to the database and automatically follows it as the current user.

```bash
gator addfeed <name> <url>
```

Example:

```bash
gator addfeed "Hacker News" "https://hnrss.org/frontpage"
```

**List all feeds**

Shows every feed in the database along with the name of the user who added it.

```bash
gator feeds
```

---

### Following

**Follow an existing feed**

Follows a feed that is already in the database by its URL. Use `gator feeds` to find URLs.

```bash
gator follow <url>
```

**List feeds you follow**

Lists every feed the current user is following.

```bash
gator following
```

**Unfollow a feed**

Removes the follow relationship by feed URL. The feed itself remains in the database.

```bash
gator unfollow <url>
```

---

### Aggregation and Browsing

**Start the aggregation loop**

Scrapes feeds continuously at the specified interval. The interval is a Go duration string. The program picks whichever feed has the oldest `last_fetched_at` timestamp each tick.

```bash
gator agg <interval>
```

Examples:

```bash
gator agg 30s      # scrape every 30 seconds
gator agg 5m       # scrape every 5 minutes
gator agg 1h       # scrape every hour
```

This command runs indefinitely. Stop it with `Ctrl+C`. Run it in a separate terminal or as a background process while you use other commands.

**Browse posts**

Prints the most recent posts from feeds you follow. Defaults to 2 posts.

```bash
gator browse [limit]
```

Example:

```bash
gator browse 10
```

## Database Schema

```
users
  id           UUID  PK
  created_at   TIMESTAMP
  updated_at   TIMESTAMP
  name         TEXT  UNIQUE

feeds
  id               UUID  PK
  created_at       TIMESTAMP
  updated_at       TIMESTAMP
  name             TEXT
  url              TEXT  UNIQUE
  user_id          UUID  -> users.id (CASCADE DELETE)
  last_fetched_at  TIMESTAMP  (nullable)

feed_follows
  id          UUID  PK
  created_at  TIMESTAMP
  updated_at  TIMESTAMP
  user_id     UUID  -> users.id (CASCADE DELETE)
  feed_id     UUID  -> feeds.id (CASCADE DELETE)
  UNIQUE (user_id, feed_id)

posts
  id            UUID  PK
  created_at    TIMESTAMP
  updated_at    TIMESTAMP
  title         TEXT
  url           TEXT  UNIQUE
  description   TEXT  (nullable)
  published_at  TIMESTAMP  (nullable)
  feed_id       UUID  -> feeds.id (CASCADE DELETE)
```

Cascading deletes are set on all foreign keys, so removing a user removes their feeds and follows. Removing a feed removes its follows and its posts. The unique constraint on `posts.url` prevents duplicate entries when the same post appears in a feed across multiple scrape cycles.

## Regenerating Database Code

If you modify any file under `sql/queries/`, regenerate the Go code with:

```bash
sqlc generate
```

The output is written to `internal/database/`. Do not edit those files by hand; they will be overwritten.
